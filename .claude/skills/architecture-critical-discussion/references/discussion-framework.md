# 批判的議論フレームワーク 詳細リファレンス

## 4軸分析の詳細手法

### 軸1: 現状分析

現状を正確に把握するための手順：

**事実と解釈の分離**
- 事実: 観測・計測・文書化されたもの（「2020年以降、空き家率が15%を超えた」）
- 解釈: 事実から導いた判断（「空き家問題が深刻化している」）
- 議論の混乱を防ぐため、両者を明示的に分ける

**現状記述のチェックリスト**
- [ ] いつの時点の状況か（時制の明示）
- [ ] どの地域・スケールの話か（スコープの明示）
- [ ] 出典・根拠はあるか（根拠の明示）
- [ ] 比較基準はあるか（何と比べて現状を評価しているか）

### 軸2: 問題の特定と構造化

問題を単なる列挙ではなく、構造として把握する。

**問題の3層構造**
```
表層問題 (Symptom)
  ↑ 引き起こされる
中間問題 (Problem)
  ↑ 引き起こされる
根本原因 (Root Cause)
```

例:
```
表層: 建物の老朽化が進んでいる
  ↑
中間: 維持管理コストが高く、補修が後回しにされる
  ↑
根本: 建物の価値が土地に吸収されてしまう日本の不動産評価システム
```

**問題記述の形式**
```
問題: [内容]
根拠: [なぜ問題と言えるか]
影響: [誰に・何に・どの程度影響するか]
緊急性: 高/中/低（理由付き）
```

**よくある問題特定の落とし穴**
- 症状を根本原因と混同する
- 問題を解決策と混同する（「対策がない」は問題ではなく状況）
- スコープ外の問題を混入させる

### 軸3: 前提の洗い出しと分類

**前提の種類**
| 種別 | 説明 | 例 |
|---|---|---|
| 事実的前提 | 事実として扱っている情報 | 「日本の木造住宅の平均寿命は30年」 |
| 価値的前提 | 何を良しとするかの判断基準 | 「建築は長寿命であるべきだ」 |
| 因果的前提 | AがBを引き起こすという想定 | 「省エネ基準を強化すれば建設コストが上がる」 |
| 制度的前提 | 現行の制度・法規が継続するという前提 | 「建築基準法の構造規定は変わらない」 |

**前提の検証状態フラグ**
- `[検証済]`: 信頼できるソースで確認済み
- `[未検証]`: 一般的に信じられているが根拠未確認
- `[要調査]`: 議論の結論に大きく影響するため調査が必要
- `[論争中]`: 複数の見解があり、確定していない
- `[仮定]`: 議論のために仮置きしているが事実とは限らない

**前提の優先順位付け**
調査リソースは有限なため、以下の基準で「要調査」前提を優先する：
1. 議論の結論への影響度（大きいほど優先）
2. 反証された場合の議論への影響（致命的ほど優先）
3. 検証の容易さ（容易なものから着手）

### 軸4: アクション導出

**アクションの分類**
| 分類 | 説明 |
|---|---|
| 即時アクション | 現状の制約内で今すぐできること |
| 中期アクション | 制度・環境の変化を前提にしたもの |
| 根本的アクション | 前提・システムそのものを変えること |

**アクション評価マトリクス**
各アクションを以下で評価する：
```
実現可能性: 高/中/低（理由）
効果: 高/中/低（どの問題に対して）
副作用: [想定されるネガティブな影響]
前提条件: [このアクションが有効になる条件]
トレードオフ: [何を得て何を失うか]
```

## 反論・批判手法

### 反論の構造

有効な反論は3要素で構成する：
```
[反論の主張] + [なぜそう言えるか（根拠）] + [元の主張のどこが問題か（論点）]
```

弱い反論の例:「それは違う」「現実的ではない」（根拠と論点がない）

強い反論の例:「省エネ基準の強化がコスト増に直結するという前提は、ドイツの事例（Passivhaus普及後のコスト変化）と矛盾する。技術革新によりコストは低減する可能性がある。」

### 論理的ギャップの発見

以下のパターンで議論の穴を探す：

**一般化の誤り**
- 「A事例でうまくいったから日本でもうまくいく」
- 確認: 事例の条件（気候、制度、文化、規模）が比較対象と同じか

**因果の逆転**
- 「建築の質が低いから住宅の価値が落ちる」↔「住宅の価値が落ちるから建築の質への投資が減る」
- 確認: 方向性の根拠を問う

**選択バイアス**
- 成功事例しか参照していない
- 確認: 失敗事例・反例は何か

**時代錯誤**
- 過去の研究・データを現在の議論に適用する
- 確認: データの時制と現在の状況の差異

**スケール問題**
- 個別事例から全体傾向を導く
- 確認: サンプルの代表性

## 知識提供と議論の接続

Phase 2で知識を提供した後、その知識を議論に活かすための接続手順：

1. 提供した知識を「前提リスト」に追加する（`[検証済]` フラグ付き）
2. その前提が4軸分析のどこに影響するかを示す
3. 「この前提を踏まえると、先ほどの問題提起はどう変わりますか？」とユーザーに問い返す

## 議論の収束手法

議論が発散した場合に収束させる手順：

1. **現在地の確認**: 「ここまでで確認できた前提・問題・アクション候補をまとめます」
2. **未解決の論点の列挙**: 「まだ答えが出ていない点は〇〇です」
3. **優先順位の確認**: 「次に集中すべき論点はどれですか？」
4. **収束判断**: ゴールに対して十分な議論ができたか確認し、結論を出す
