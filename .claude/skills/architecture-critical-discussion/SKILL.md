# Architecture Critical Discussion

批判的思考と構造化された議論フレームワークを用いて、建築テーマの議論を深めるskill。議論は4つのフェーズで進む。前提が不足している場合は議論を中断し、`architecture-research` skillをサブエージェントとして起動して調査を行う。

## 議論の4フェーズ

### Phase 1: 方針確認

議論を開始する前に、必ずユーザーへ方針を確認する。以下の点を一度に聞きすぎず、最重要事項から始める：

**確認すべき項目（優先順）：**
1. 議論のゴール — 何を明らかにしたいか / 何を決めたいか
2. スコープ — 対象範囲（時代、地域、建築種別、技術領域など）
3. 優先事項 — 批判的分析 / 代替案の探索 / 前提の検証 / アクション導出、のどれを重視するか

**方針確認の判断基準：**
- ユーザーの指示が具体的であれば、確認を省略して Phase 2 へ進む
- 指示が曖昧 / 広すぎる場合は、上記3点を確認してから進む

### Phase 2: 知識アセスメント

ユーザーの理解レベルを確認し、必要に応じて背景知識を提供する。これを怠ると、後続フェーズでのユーザーへの指示や確認が機能しなくなる。

**アセスメントの方法：**
- ユーザーが使う語彙・説明の具体性からレベルを推定する
- 不確かな場合は「〇〇については既にご存知ですか？」と直接確認する

**知識提供のトリガー：**
- 議論に必須の概念をユーザーが誤解している
- 議論の前提となる制度・法規・歴史的背景に言及がない
- ユーザーの前提が事実と異なる可能性がある

**知識提供の原則：**
- 過不足なく提供する（詳細は `references/knowledge-assessment.md` を参照）
- 知識提供後、ユーザーの理解を確認してから次フェーズへ進む

### Phase 3: 批判的構造化分析

以下の4軸で議論を構造化する。各軸は独立して分析した後、相互の関係を明確にする。

**分析の4軸：**

| 軸 | 問い | 出力形式 |
|---|---|---|
| 現状 | 今何が起きているか？ | 事実の列挙（出典を意識） |
| 問題 | 何が問題か？なぜ問題か？ | 問題 → 根拠 → 影響の連鎖 |
| 前提 | 暗黙に仮定されていることは何か？ | 前提リスト + 検証状態のフラグ |
| アクション | 何をすべきか？何が可能か？ | 選択肢 + トレードオフ |

**批判的分析の手順：**
1. ユーザーの主張・問いを一文で言い換えて確認する
2. 4軸それぞれを分析する
3. 前提リストを作成し、「検証済み」「未検証」「要調査」をフラグ付けする
4. 問題とアクションの対応関係を明示する
5. 議論の穴（論理的ギャップ、反論の余地）を提示する

詳細な分析フレームワークは `references/discussion-framework.md` を参照。

### Phase 4: 調査トリガー

前提の検証状態が「要調査」になった場合、または調査をユーザーが明示的に指示した場合に発動する。

**自動トリガーの条件：**
- 議論の核心となる前提が検証されていない
- 事実として提示された情報に根拠がない
- 数値・法規・事例の正確性が議論の結論を大きく左右する

**トリガー時の手順：**
1. 議論を中断し、「〇〇の前提を調査する必要があります」と明示する
2. 調査対象と調査の目的を明確にする
3. `architecture-research` skillをサブエージェントとして起動する：
   ```
   Agent tool を使用し、architecture-research skill の手順に従って以下を調査:
   - 調査対象: [具体的な前提・事実]
   - 調査目的: [議論との関連性]
   - 必要な情報: [具体的に何が分かればよいか]
   ```
4. 調査結果を受け取り、前提リストのフラグを更新する
5. 更新された前提を踏まえて議論を再開する

**ユーザー指示トリガーの場合：**
- 「調査して」「確認して」「根拠を調べて」などの指示を検出する
- 調査スコープをユーザーと確認してから起動する

## 議論の継続と収束

各フェーズを経た後、以下のいずれかで議論を進める：

- **深化**: 特定の問題軸・前提をさらに掘り下げる
- **拡張**: 関連する別の前提・問題軸に展開する
- **収束**: アクションの優先順位を決定し、議論を結論へ導く

ユーザーが次の方向を明示しない場合は、選択肢を提示して判断を求める。

## 議論の記録形式

各フェーズの成果を以下の形式で整理する：

```
## 議論サマリー
**ゴール:** [方針確認で合意したゴール]
**スコープ:** [対象範囲]

## 前提リスト
- [前提A] — 検証済み / 未検証 / 要調査
- [前提B] — ...

## 問題構造
- 問題1: [内容] → 根拠: [根拠] → 影響: [影響]

## アクション候補
1. [アクション] — メリット/デメリット
2. ...

## 議論の穴・残課題
- [未解決の論点]
```
