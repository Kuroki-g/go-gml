# Architecture Research

建築テーマの調査を構造化されたアプローチで実施するskill。WebFetch単体では対応しきれない多段階・複数ソース調査を、WebSearch + WebFetch の組み合わせで実現する。

## なぜ WebFetch 単体では不十分か

- 単一URL固定: 検索が必要な場合に対応不可
- 日本語学術ソース（J-STAGE、CiNii）はナビゲーション複数ステップが必要
- 建築基準法等の法令は複数条文の横断参照が必要
- 事例調査は複数サイト・複数ページの統合が必要

WebSearch で対象を特定し、WebFetch で精読する2段階アプローチを基本とする。

## 調査の基本フロー

```
1. 調査設計     — 調査対象と必要情報を明確化
2. 広域探索     — WebSearch で候補ソースを列挙
3. ソース評価   — 信頼性・関連性でフィルタリング
4. 精読         — WebFetch で必要ページを詳細取得
5. 統合・検証   — 複数ソースの照合・矛盾の検出
6. 報告         — 構造化された調査結果を返す
```

## Step 1: 調査設計

調査を開始する前に、以下を明確にする：

- **調査対象**: 何の前提・事実・事例を調べるか
- **必要な情報**: どの情報が分かれば議論に使えるか
- **精度要件**: 概略把握でよいか / 詳細な数値・出典が必要か
- **言語スコープ**: 日本語のみ / 英語も含む

`architecture-critical-discussion` からの起動の場合、呼び出し元が調査対象と目的を渡す。

## Step 2: 広域探索 (WebSearch)

調査対象に応じてWebSearchクエリを設計する。

**クエリ設計の原則：**
- 日本語ソースは日本語クエリを使う（機械翻訳ではなく自然な専門語）
- 汎用クエリと専門クエリを組み合わせる
- サイト指定（`site:e-gov.go.jp` など）で公式ソースを優先する

**領域別の推奨クエリパターン:**

| 領域 | 推奨クエリ例 | 優先ソース |
|---|---|---|
| 建築基準法・法令 | `建築基準法 第〇条 site:e-gov.go.jp` | e-Gov法令検索 |
| 学術論文 | `[テーマ] 建築学会 OR J-STAGE` | J-STAGE, CiNii |
| 国内事例 | `[テーマ] 建築事例 設計` | 建築学会、専門誌 |
| 海外事例 | `[テーマ] architecture case study` | 英語学術サイト |
| 統計・データ | `[テーマ] 統計 国土交通省` | 国土交通省、総務省 |
| 歴史的背景 | `[テーマ] 歴史 変遷 建築` | 学会論文、書籍引用 |

複数クエリを試して、結果の質を比較する。

## Step 3: ソース評価

WebSearchで得られたURLを以下の基準でフィルタリングする：

**信頼性評価（高→低）:**
1. 政府・行政機関（e-gov.go.jp, mlit.go.jp, 各都道府県）
2. 学術機関・学会（J-STAGE, CiNii, 大学リポジトリ）
3. 専門業界団体（日本建築学会、日本建築士会連合会）
4. 専門誌・業界誌（建築雑誌、日経アーキテクチュアなど）
5. その他（一般サイト、個人ブログ）

**関連性評価:**
- 調査対象に直接関連するか
- 情報の新しさ（建築基準は改正日を確認）
- 一次情報か二次情報か（一次情報を優先）

評価結果を記録し、精読対象を3〜5件に絞る。

## Step 4: 精読 (WebFetch)

選定したURLをWebFetchで取得し、必要情報を抽出する。

**精読の手順:**
1. 目的に関連するセクションを特定する
2. 数値・年号・固有名詞は正確に引用する（要約ではなく原文抽出）
3. 引用元情報（URL、取得日、著者・機関名）を記録する
4. 1ページで情報が不足する場合、リンクを辿って追加取得する

**複数ページが必要な場合:**
- 法令の場合: 条文本文 + 施行令・省令も確認
- 学術論文の場合: 要旨 → 必要なら本文
- 事例の場合: 概要ページ + 詳細ページの両方

## Step 5: 統合・検証

複数ソースの情報を照合し、矛盾や不確実性を明示する。

**照合の手順:**
1. 同一事実に対して複数ソースの記述を並べる
2. 一致する部分を「確認済み」とする
3. 矛盾・差異がある部分を明示し、どのソースが信頼性が高いか判断する
4. 情報が見つからなかった場合は「調査範囲内では確認できず」と明示する

**確認できなかった情報の扱い:**
- 「調査範囲内では根拠が見つからない」と明記する
- 追加調査が必要か、別の調査アプローチが必要かを示す

## Step 6: 調査結果の報告

`architecture-critical-discussion` への報告、またはユーザーへの直接報告に使う構造化フォーマット：

```markdown
## 調査結果

**調査対象:** [何を調査したか]
**調査目的:** [議論との関連]

### 確認された事実
- [事実1] — 出典: [URL / 機関名 / 発行年]
- [事実2] — 出典: ...

### 未確認・不確実な情報
- [事項] — 理由: [なぜ確認できなかったか]

### ソース一覧
1. [機関名/サイト名] — [URL] — 信頼性: 高/中/低
2. ...

### 議論への示唆
- [調査結果が議論の前提・結論にどう影響するか]

### 追加調査が必要な場合
- [何をさらに調べると議論が深まるか]
```

## 調査スコープの判断

**1回の調査セッションで扱う範囲:**
- 明確な前提1〜3件を深く調べる（広く浅くではなく）
- 調査時間と精度のバランスを取る

**追加調査が必要な判断基準:**
- 核心的前提が確認できなかった
- 矛盾するソースが見つかった
- 調査中に新たな重要前提が発見された

追加調査の必要性をユーザーまたは呼び出し元のskillに伝える。
